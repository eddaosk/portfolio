# Brainstorming Session Results

**Facilitator:** Edda
**Date:** 2026-05-17

## Session Overview

**Topic:** Vasuqi ApS website — visual identity, design system, and AI-agent design manual for handoff
**Goals:** Generate ideas for the coded website and the design spec/manual that will guide AI agents as the owner continues to build on top of the delivered codebase

### Session Setup

Edda is a multimedia design student working on her final project: building and delivering a coded website for Vasuqi ApS, a Danish clean-tech startup (DTU spinout) that makes visible-light photocatalytic water purification systems. The team has completed logo, brand identity, color palette, and a Figma prototype. Now the task is to code it and deliver it with a design manual that lets AI coding agents maintain visual consistency as the owner vibe-codes new features on top.

## Phase 1: What If Scenarios — Ideas Generated

### Voice & Copy Rules

**[Manual #1]: Color System with Usage Rules**
_Concept:_ Not just the palette, but which colors are primary load-bearers, which are accent-only, and explicit rules for when each gets used — e.g. "electric blue only for interactive elements or technology highlights, never as a background."
_Novelty:_ AI agents default to using all brand colors equally. Rules prevent that drift.

**[Manual #2]: Emotional Intent**
_Concept:_ The feelings the brand should evoke — trust, precision, quiet innovation — written explicitly as a target state, not just adjectives.
_Novelty:_ Most design docs describe visuals. This tells the AI _why_ the visuals exist, which lets it make judgment calls on new content.

**[Manual #3]: Aesthetic Vocabulary**
_Concept:_ Minimalism as a defined direction — but with guardrails. What KIND of minimalism? Clinical? Warm? Cold? Hard-edged? Vasuqi's is "Hard Tech Minimalist" — and needs to be named and described.
_Novelty:_ "Minimalism" means different things in different contexts. Naming the specific variant prevents generic minimalism.

**[Manual #4]: Brand Voice — The "Stupidly Simple" Principle**
_Concept:_ Copy is direct, confident, never over-explained. The solution is light — that's it. Headlines are short. No filler words. No corporate hedging. The brand owns its simplicity rather than apologising for it.
_Novelty:_ Most tech companies over-explain their complexity. Vasuqi does the opposite. An AI needs the rule: simplicity is the argument, not the disclaimer.

**[Manual #5]: Tone Polarity — Light Not Dark**
_Concept:_ The brand is positive, clean, calm. Mood is not dystopian or urgent. Even when talking about water pollution and industrial failure, the framing is forward-looking — "here's what becomes possible," not "here's what's broken."
_Novelty:_ This cuts against the cyberpunk direction deep-tech brands often take. Without this rule the tone drifts heavy.

**[Manual #6]: Water as Aesthetic Logic**
_Concept:_ Clean, calm, pure — these aren't just adjectives, they're the conceptual source of design decisions. White space, clean typography, absence of clutter all trace back to water. The AI needs to know _why_, not just _what_.
_Novelty:_ When an AI knows the reasoning, it can apply it to new situations it wasn't trained on.

**[Manual #7]: Headline Formula — Short Claim, Then Evidence**
_Concept:_ Every section follows the same structure: punchy 3–5 word header, then one data sentence, then context. Never the other way around. Lead with the claim, earn it after.
_Novelty:_ An AI writing copy tends to lead with context and bury the claim. This reversal is the brand's signature rhythm.

**[Manual #8]: The Gradient Narrative**
_Concept:_ The page moves dark → light as you scroll. The hero is deep navy (the problem, murky water) and the page physically brightens toward solution. That's a spatial argument, not just aesthetics.
_Novelty:_ New sections must live within this gradient logic, not break it.

**[Manual #9]: The Anti-Hype Voice**
_Concept:_ "No chemicals added. No sludge generated." — short, negative-space statements that prove simplicity by listing what's absent. The brand builds trust by telling you what it doesn't do.
_Novelty:_ Most marketing adds. Vasuqi subtracts. That's a copywriting rule.

**[Manual #10]: Positioning Over Competing**
_Concept:_ "Don't replace what works. Finish what conventional treatment can't." — Vasuqi never attacks competitors directly. It claims the gap they leave.
_Novelty:_ An AI might write "better than ozonation." The rule is: never compare, only complete.

### Material Languages & Visual System

**[Manual #11]: Water-Form Shape Language**
_Concept:_ Organic, circular, fluid shapes are the brand's native container for mission-critical content — not rectangular cards. Rule: when grouping items connected to Vasuqi's core purpose, use water-form shapes. Rectangular cards are for utility content (specs, data tables).
_Novelty:_ Without this rule, an AI defaults to rectangles — the internet's default.

**[Manual #12]: Shape → Meaning Map**
_Concept:_ Circles/orbs = core brand values, mission, what water means. Rectangles with rounded corners = technical/informational content. Sharp rectangles = never.
_Novelty:_ Gives the AI a decision tree for shape choice, not just examples to copy.

**[Manual #13]: The Three Material Languages**
_Concept:_ Every visual decision traces back to one of three materials — **Water** (fluid, clean, organic), **Glass** (transparency, blur, iridescence, frosted luxury), or **Blueprint** (line-drawn, technical precision, unrealised potential). These are the decision filters above the component level.
_Novelty:_ Most design systems define components. This defines the reasoning layer above them — which is what an AI needs to make judgment calls on novel situations.

**[Manual #14]: Blueprint = Line, Not Fill**
_Concept:_ Icons, diagrams, and technical illustrations are always line-drawn, never filled. Conceptually tied to the product's status — a blueprint of something being built. Filling the icons would claim a certainty the product doesn't yet have.
_Novelty:_ A visual rule with a direct conceptual argument attached. "Use outline icons because the product is in blueprint stage" — so the rule survives even when the next designer doesn't know the backstory.

**[Manual #15]: Glass as Luxury Transparency**
_Concept:_ The glass material language has four qualities — **clear/see-through** (transparency doctrine), **iridescent** (light through water), **frosted** (soft, not harsh), **polished/luxurious** (precision hardware, not budget industrial). Each quality maps to a brand value.
_Novelty:_ "Glassmorphism" as a trend label doesn't capture this. The manual names all four qualities and explains why they coexist.

**[Manual #16]: Material → Element Mapping**
_Concept:_ Glass applies to UI containers — cards, section backgrounds, overlays, navigation. Blueprint applies to representational elements — icons, technical diagrams, product illustrations. They don't overlap.
_Novelty:_ Without this mapping, an AI might apply glass effects to icons and break the system.

### Design Evolution & Audience

**[Manual #17]: Blueprint Phase Declaration**
_Concept:_ The current visual identity is explicitly Phase 1 — investor-facing, pre-prototype. The manual should include a version marker: "Design System v1.0 — Investor Phase." Line-drawn icons and blueprint aesthetic are intentional choices for this stage.
_Novelty:_ Documenting a design system's lifecycle stage tells the next designer that this system has a known expiry condition, not a flaw.

**[Manual #18]: Evolution Trigger Conditions**
_Concept:_ Define when the visual identity should evolve — not a date, but a product milestone. Triggers: working prototype exists → introduce real product photography. Pilot programs launch → shift from investor tone to B2B operator tone. The blueprint aesthetic is retired intentionally, not abandoned.
_Novelty:_ Most design systems document what the brand IS. This one also documents what it becomes, and why.

**[Manual #19]: Fixed vs. Flexible System**
_Concept:_ Two layers. **Fixed** (never changes per audience): colour palette, typography, glass/water/blueprint material language, shape vocabulary, tone of voice. **Flexible** (adapts per audience): which pain points are foregrounded, which industry terminology is used, which numbers are cited.
_Novelty:_ Without this rule, an AI either over-adapts (breaks the brand per page) or under-adapts (writes the same copy for everyone).

**[Manual #20]: Necessity-First Copy Structure**
_Concept:_ Every page, every section, every audience follows the same argumentative logic: (1) establish the problem as undeniable and expensive, (2) show that existing solutions fail at this specific gap, (3) present Vasuqi as built precisely for this. Never lead with the product. Lead with the pain.
_Novelty:_ This is a content architecture rule, not just a tone rule — directly arguable at exam as a persuasion strategy (problem → conflict → resolution).

**[Manual #21]: The Necessity Anchor**
_Concept:_ One claim is constant across all copy, all pages, all audiences: the product exists because the regulatory and economic pressure is undeniable — "the targets are set." This line (or its equivalent) must appear on every page in some form.
_Novelty:_ Gives the AI an invariant — a sentence it cannot remove or soften regardless of what else changes.

### Responsive Behaviour

**[Manual #22]: Responsive Grid Rules**
_Concept:_ Desktop uses 4-column and 3-column grids. Mobile collapses to 2-column for card groups, single-column for narrative sections. White space compresses but has a floor — the calm/clean feeling must survive on small screens.
_Novelty:_ Not just "how many columns" but a rule that whitespace has a minimum — the AI can compress but cannot eliminate it.

**[Manual #23]: Flag-for-Human Elements**
_Concept:_ Some elements can't be responsively adapted by rule alone. The data visualisation graphs (COD/cost line charts) are the primary example — they may need to be redesigned on mobile, not just scaled. The manual explicitly lists these as "do not auto-adapt — flag for review."
_Novelty:_ Teaching the AI what it can't decide is as important as teaching it what it can.

### Typography

**[Manual #24]: Typography Pairing Logic**
_Concept:_ **Syne** (headings) — geometric, avant-garde, slightly technical. **Manrope** (body/UI) — clean, calm, highly readable. Syne carries the blueprint/innovation language, Manrope carries the water/calm language. The pairing maps onto the brand's core tension: bold technology communicated with calm clarity.
_Novelty:_ The font choice is an extension of the material system, not just aesthetics.

**[Manual #25]: Typography Hierarchy Rules**
_Concept:_ Syne is reserved exclusively for headings and display text. Manrope handles everything else — body copy, captions, navigation, UI labels, CTAs. Rule is absolute: never Syne for body, never Manrope for primary headings.
_Novelty:_ Without this rule, an AI "harmonising" the design starts using Manrope everywhere and erases the brand's typographic voice.

**[Manual #26]: Complete Typography Scale**
_Concept:_ Five levels, precisely defined:

| Level | Font | Weight | Size | Line Height |
|---|---|---|---|---|
| H1 | Syne | 743 (variable) | 55pt | 106% |
| H2 | Syne | SemiBold (600) | 24pt | Auto |
| H3 | Syne | Medium (500) | 20pt | Auto |
| Body | Manrope | Medium (500) | 16pt | Auto |
| Caption | Manrope | ExtraLight (200) | 20pt | Auto |

_Novelty:_ Exact values must be documented — especially H1's variable weight of 743. Coding it as `font-weight: 700` would look visibly different.

**[Manual #27]: The Counter-Intuitive Caption Rule**
_Concept:_ Captions are 20pt ExtraLight — physically larger than body text (16pt) but much lighter in weight. They feel ghostly, atmospheric. Large but weightless connects to the glass/ethereal material language.
_Novelty:_ This is the rule an AI will most likely "correct" away. Every design instinct says captions should be small. Here they are large and feather-light.

**[Manual #28]: Variable Font Declaration**
_Concept:_ Syne is loaded as a variable font. H1 uses weight 743 — a value between SemiBold and Bold that only exists on the variable axis. In CSS: `font-weight: 743`. Any implementation must load the variable font file, not a static weight subset.
_Novelty:_ A developer or AI installing "Syne Bold" gets the static 700 weight and loses the H1's specific character. Silent, invisible mistake.

### Colour System

**[Manual #29]: Complete Color Palette with Roles**
_Concept:_ Ten values, each with a defined role:

| Token | Hex | Role |
|---|---|---|
| `--navy-deep` | `#0A1F44` | Primary dark background, hero, footer |
| `--blue-primary` | `#0044FF` | Main accent — buttons, active states, logo |
| `--blue-deep` | `#0033CC` | Secondary accent — hover states, depth |
| `--blue-mid` | `#6A93FF` | Mid-tone — glass layers, secondary UI |
| `--blue-soft` | `#A8C5FF` | Light accent — gradient mid-stops, hover tints |
| `--cyan-light` | `#00E5FF` | Technology highlight — LED glow effect only |
| `--steel` | `#5C6B85` | Neutral — body text on light, secondary labels |
| `--ice-light` | `#D6F8FF` | Light section backgrounds |
| `--ice-near` | `#E8F2FF` | Near-white sections |
| `--white-brand` | `#FAFCFF` | Brand "white" — never pure white |

_Novelty:_ The palette is the technology made visible. `#00E5FF` cyan is the LED glow colour. `#0A1F44` is the depth of unclean water. The colour system is a narrative, not decoration.

**[Manual #30]: The No-Pure-Black, No-Pure-White Rule**
_Concept:_ `#000000` and `#FFFFFF` are permanently banned. Darkest value is `#0A1F44`. Lightest is `#FAFCFF`. Pure black is industrial; pure white is clinical in the wrong direction.
_Novelty:_ This is the rule an AI will break first. It defaults to `#000000` for text and `#FFFFFF` for backgrounds.

**[Manual #31]: Palette Closure Rule**
_Concept:_ The AI must never introduce colours outside the ten-value palette. If a use case feels like it needs a new colour, that's a signal the component needs redesigning, not that the palette needs expanding.
_Novelty:_ Explicitly naming this as a hard constraint prevents slow palette drift over many AI-assisted updates.

**[Manual #32]: Motion = Water Physics**
_Concept:_ All animations are governed by how water moves — slow, continuous, non-linear, never sudden. Entrance effects use long durations (800ms–1200ms), gentle ease-in-out curves. Background motion is continuous and nearly imperceptible. No bouncing, no spring physics, no snap, no flash.
_Novelty:_ "Water physics" as the motion reference gives the AI a conceptual test for any new animation: does this feel like water moving? If not, reject it.

**[Manual #33]: Cyan is LED-Only**
_Concept:_ `#00E5FF` is reserved exclusively for light-source moments — glow effects, the product's LED highlight, technology-specific accents. Never as a background, text colour, or generic accent. It means: this is where the light comes from.
_Novelty:_ The brightest colour in the palette is also the most restricted. Without this rule, an AI treats it as a general-purpose highlight and it loses its meaning.

**[Manual #34]: Logo Mythology — Do Not Alter**
_Concept:_ The logo contains Vasuki (Hindu serpent king) forming a circle — the water cycle, purification, transformation. Internal waves use three tones (`#0044FF`, `#6A93FF`, `#A8C5FF`) representing water getting progressively cleaner. This is a fixed asset — colours, proportions, and layering are never modified.
_Novelty:_ Documenting the mythology behind the colour layers makes clear this is a meaning-bearing system, not decoration.

**[Manual #35]: Entrance Animation Concept**
_Concept:_ Planned entrance sequence: text builds first ("The last 5%... is the hardest") → resolution line ("Water, polished by light") → logo animates in. The sequence follows the brand's argumentative logic — problem first, solution second, identity last.
_Novelty:_ The animation sequence IS the brand argument. Any future entrance animation should follow this logic.

### Anti-List

**[Manual #36]: The Generic Trap — Anti-Rule**
_Concept:_ The most dangerous outcome is a page that looks professionally competent but brand-neutral — a page that could belong to any deep-tech company. If someone can't tell it's Vasuqi within two seconds, it has failed.
_Novelty:_ Names the specific failure state — "generic" — and makes it a measurable outcome, not just an aesthetic preference.

**[Manual #37]: No Dark Mode. Ever.**
_Concept:_ Dark mode is explicitly forbidden. AI defaults to dark mode for deep-tech brands. Vasuqi is the opposite argument — light, clean, water, purity. Dark mode breaks the conceptual logic: clean water is not dark.
_Novelty:_ Counter-intuitive decision with a clear rationale: "We chose light because our product makes things light. Dark mode would be a lie."

**[Manual #38]: No AI-Default Gradients**
_Concept:_ Gradients may only use colours from the ten-value palette and must travel in the dark-to-light direction — never sideways, never dark-to-darker. AI-generated gradients default to generic purple-to-blue or arbitrary transitions.
_Novelty:_ Direction matters. A gradient going from `#0A1F44` to `#FAFCFF` is the brand argument. The same colours reversed are decorative noise.

**[Manual #39]: Use Designed Assets, Not Code Approximations**
_Concept:_ Custom icons, the logo, the water background animation are designed artefacts — not CSS effects to be recreated. If an icon doesn't exist in the icon set, request it. Don't substitute a generic icon library. Don't approximate the water animation with a CSS gradient pulse.
_Novelty:_ Code that mimics design is always detectable and always looks cheaper. The AI implements the design system, it does not invent workarounds for it.

**[Manual #40]: No Generic Card Components**
_Concept:_ The default AI card (white rectangle, rounded corners, drop shadow, icon, title, body text) is banned. Vasuqi's content containers are water-form shapes or glassmorphism panels. If content needs a container not in the design system, use the closest defined pattern — never invent a generic card.
_Novelty:_ "Generic card" is so default it's almost invisible. Naming it as banned forces the AI to actively choose a brand-appropriate alternative.

**[Manual #41]: Navigation Must Stay Simple**
_Concept:_ Current navigation has three items plus a contact CTA. The AI must not add dropdowns, mega-menus, nested navigation, or secondary nav bars. If the site grows, pages are added to the existing structure, not layered beneath it.
_Novelty:_ AI over-structures navigation as sites grow. Connects directly to the "stupidly simple" voice principle — navigation is also communication.

## Phase 2: Six Thinking Hats — Complete

**[Manual #42]: Every Rule Has a Three-Layer Argument**
_Concept:_ Each rule traces to: (1) a design principle or theory, (2) one of the three material languages, (3) research or testing. One layer = preference. Three layers = decision.
_Novelty:_ Makes the manual exam-proof and more persuasive to the owner — he follows rules he understands.

**[Manual #43 — Black Hat]: The Component Gap**
_Concept:_ The current design system covers what exists. It doesn't equip an AI to invent new components. When the site grows, the AI will default to generic patterns without component-level guidance. Named as a known limitation of v1.0.
_Novelty:_ Naming limitations honestly sets correct expectations and prevents silent failures.

**[Manual #44]: Component Derivation Protocol**
_Concept:_ When a component doesn't exist: (1) Which material language — water, glass, or blueprint? (2) What's the closest existing component? (3) Container needed? Water-form if mission-critical, glass panel if informational. (4) Icon? Line-drawn only. (5) Colour? Text in `#0A1F44`, accents in `#0044FF` for interactive states only.
_Novelty:_ Gives the AI a derivation method, not just a reference library. The system can grow without the designer.

**[Manual #45]: Media Asset Policy**
_Concept:_ Ambient video (water footage): keep until a product video exists. Product visualization (blueprint render): replace only with real product photography. Team photography: exempt from colour palette rule. All other new imagery: cool tones, clean composition, no warm colours, no busy backgrounds.
_Novelty:_ Naming the team photo exception prevents the AI from trying to colour-grade portraits blue.

**[Manual #46]: The North Star Rule**
_Concept:_ One sentence governing everything: "If anything new is added, it must fit the brand identity, visual identity, and tone of voice." When no specific rule applies, this is the test.
_Novelty:_ A master override clause that covers every gap in every other rule.

**[Manual #47]: Accessibility as Ongoing Requirement**
_Concept:_ Current palette verified against WCAG. Any new text colour combination must be contrast-checked before implementation. Light-on-light palette combinations are for decorative elements only — never text.
_Novelty:_ Frames accessibility as a forward requirement, not just a current state.

**[Manual #48]: Design Manual Format — Two Files**
_Concept:_ `docs/design-manual.md` for rules and reasoning (what BMad agents read for judgment calls) + `design-tokens.css` for all values as CSS custom properties (what gets copied into code). The markdown has the why; the CSS has the what.
_Novelty:_ Separating reasoning from implementation means the AI can reference the manual for decisions AND use tokens for precision without parsing prose for hex codes.

**[Manual #49]: Design Manual Length — 400–500 Lines**
_Concept:_ The markdown manual must not exceed 400–500 lines — comprehensive enough to cover all decisions, short enough for a BMad agent to process fully in one context window. Every rule must earn its place.
_Novelty:_ Length is a design decision. A manual that fits in context is always consulted. One that doesn't gets skimmed.

**[Manual #50]: Design Manual Structure — Task-Based**
_Concept:_ Organised by what the AI needs to decide: (1) North Star, (2) Quick Reference — tokens + anti-list, (3) When adding a new section, (4) When writing new copy, (5) When creating a new component, (6) When adding media, (7) Evolution triggers.
_Novelty:_ AI agents ask "what should I do here?" not "what category does this fall under?" Task-based structure matches how agents use documentation.

## Phase 3: First Principles Thinking — Complete

**[Manual #51]: The Manual is Authoritative, Not Advisory**
_Concept:_ The manual replaces the AI's general web design knowledge for Vasuqi decisions — it is not a supplement. If something isn't in the manual, the AI flags it rather than invents from internet defaults.
_Novelty:_ States this explicitly at the top of the document so the AI reads everything that follows with that frame.

**[Manual #52]: The Extension Principle**
_Concept:_ When the manual doesn't address a situation: (1) find the closest relevant rule or component, (2) apply the three material languages as the reasoning layer, (3) build so it could have come from the same designer. Never invent from scratch.
_Novelty:_ Turns the manual from a closed catalogue into a living system with derivation logic.

**[Manual #53]: The "Same Designer" Test**
_Concept:_ Final check for anything new — does this look like it was made by the same designer as the rest of the site? Not "does this follow the rules" but "does this feel continuous." If it reads as an addition rather than an extension, revise.
_Novelty:_ Catches the cases rules miss — a holistic test, not a checklist.

**[Manual #54]: The Living Manual Protocol**
_Concept:_ Every time a new component or pattern is built, it gets documented in the manual before the session ends. Name, material language, use case, key decisions, code reference. The manual is the institutional memory of the design system.
_Novelty:_ Creates continuity across sessions and people. The carousel built in month three is the same one used in month nine.

**[Manual #55]: New Component Documentation Template**
_Concept:_ Five fields only — Name, Material Language, Use Case, Key Decisions, Code Reference. Designed for speed: if documenting takes 2 minutes it happens; if it takes 20 it doesn't.
_Novelty:_ Low friction is a design feature of the template itself.

**[Manual #56]: Dual Audience Design**
_Concept:_ The manual serves two readers — the AI agent (precise, command-style rules) and the founder (non-designer, needs plain-language explanation). Two layers, one document. The AI reads the rules; the founder reads the context around them.
_Novelty:_ If the founder doesn't understand the manual, he bypasses it. Both audiences must be served.

**[Manual #57]: Founder Onboarding Section**
_Concept:_ First section written entirely for the founder: what this document is, how to use it with an AI agent (paste as context before building anything), how BMad connects to it, how to document new components. Plain language, under one page.
_Novelty:_ The manual is useless if the founder doesn't know how to activate it.

**[Manual #58]: Jargon Glossary**
_Concept:_ Short glossary at the end defines design terms used in the manual — glassmorphism, variable font, WCAG, material language, CSS custom properties. Removes friction between the founder reading a rule and acting on it.
_Novelty:_ Non-designer founder shouldn't need to Google terms to follow the rules.

## Phase 4: Solution Matrix — Complete

**[Manual #59]: Document Structure as Deliverable**
_Concept:_ The manual has four working parts with a clear job each: Part 1 is for the founder to read once. Part 2 is what the AI scans first when building. Part 3 is what the AI consults when the founder asks strategic questions. Part 4 is the derivation engine for growth. Parts 5–6 are maintenance and reference.
_Novelty:_ The structure is itself a design decision — it determines which rules get used and when. A flat list of 59 rules would be equally complete but far less usable.

### Final Document Structure

**File 1: `docs/design-manual.md`** (~480 lines)

```
# Vasuqi Design System — v1.0
Status: Investor Phase (pre-prototype)

── PART 1: FOR THE FOUNDER (~80 lines) ──
## What This Document Is         plain prose, ~80 words
## How to Use It With an AI      3-step numbered guide
## How to Add New Components     5-field template + filled example

── PART 2: QUICK REFERENCE — BUILD MODE (~120 lines) ──
## The One Rule (North Star)     1 bold sentence
## Never Do This (Anti-List)     12 numbered rules, 1 sentence each
## Colour Tokens                 table: token | hex | role
## Typography Scale              table: level | font | weight | size | line-height

── PART 3: BRAND FOUNDATION — GUIDE MODE (~140 lines) ──
## The Three Material Languages  Water / Glass / Blueprint — definition + applies to
## Voice & Copy Rules            8 named rules, 2 sentences each
## Motion Rules                  4 named rules
## Shape & Layout Rules          gradient direction, shape→meaning, whitespace floor

── PART 4: BUILDING NEW THINGS (~80 lines) ──
## Component Derivation Protocol 5-step IF/THEN chain
## The Same-Designer Test        1 paragraph
## Flag-for-Human Conditions     bulleted list of situations requiring founder decision

── PART 5: LIVING MANUAL (~40 lines) ──
## Documented Components         empty at handoff, grows over time
## Evolution Triggers            5 bullet conditions for when to update

── PART 6: GLOSSARY (~30 lines) ──
  10–12 defined terms
```

**File 2: `design-tokens.css`** (~60 lines)
All colour values, typography scale, and spacing as CSS custom properties — ready to paste directly into code.

### Content × Format Matrix

| Content | Format | Rationale |
|---|---|---|
| Founder onboarding | Plain prose | Human reads this, not the AI |
| How to activate with AI | Numbered steps | Actionable, sequential |
| Colour tokens | CSS code block (separate file) | Copy-paste, zero ambiguity |
| Type scale | Table | Scannable, all values visible at once |
| Anti-list | Numbered bold list | Quick-scan, memorable |
| North Star rule | Single bold sentence | Can't be missed |
| Three material languages | Named definitions, 3 sentences each | AI needs concept + application |
| Voice rules | Named rules, 1–2 sentences | Command-style, no prose padding |
| Motion rules | Named rules | Same |
| Component derivation | IF/THEN numbered steps | Decision chain, sequential |
| New component template | Fillable template block | Low friction to copy and complete |
| Evolution triggers | Bullet conditions | Clear on/off states |
| Glossary | Definition list | Fast lookup |

### Two Modes — How the AI Uses the Manual

**Build Mode** (AI is coding something new): scan Part 2 Quick Reference first, then the relevant rule in Part 3.

**Guide Mode** (founder asks "should I add X?" or "how should this feel?"): read Part 3 Brand Foundation, consult Part 5 Evolution Triggers if it involves site growth.

### What Was Established

**The manual's identity:** Authoritative, not advisory — the AI's only design reference for Vasuqi, not a supplement to general web knowledge.

**The brand system:** Three material languages (Water / Glass / Blueprint) as the reasoning layer above all rules. Ten-colour palette rooted in the technology itself. Syne + Manrope pairing mapping onto the brand's core tension. Phase 1 (Investor-facing) with documented evolution triggers.

**The deliverable:** Two files. `docs/design-manual.md` (~480 lines, task-based structure, dual audience). `design-tokens.css` (~60 lines, CSS custom properties). Together they give the AI what it needs to build and the founder what he needs to understand.

**The living system:** Component Derivation Protocol + Living Manual Protocol + Same Designer Test = the manual grows with the website and maintains continuity across sessions, founders, and AI agents.
