## What Went Well

- **Clean greenfield execution** — all 4 stories hit done with no blockers; CI pipeline live on GitHub Pages from day one.
- **Bespoke `htmlPartialsPlugin`** — zero external partial-injection dependencies; full project control, path traversal guard added in review.
- **Design token system achieved AC6** — zero hex values in any source stylesheet. One hex source of truth (`design-tokens.css`) from day one.
- **AI instruction files in place early** — `CLAUDE.md`, `.cursorrules`, `copilot-instructions.md` set up the founder handoff mindset from Epic 1.
- **Prototype preservation** — `_prototypes/` folder keeps earlier exploration work (water droplet, logo animation) accessible for Epic 2 reference.
- **Upfront investment paid off** — brainstorming, architecture, and Figma file as reference all reduced ambiguity during implementation. For a first-time framework user, this was the right call.

---

## Challenges & Growth Areas

- **The 'nothing looks like anything' gap** — Epic 1 built infrastructure (tokens, pipeline, shells) with no visual payoff. This was expected and correct, but disorienting for a designer-first project lead used to seeing immediate visual feedback. Future epics should be framed with explicit 'what you'll see when this is done' language.
- **Bootstrap-to-Tailwind transition** — Tailwind v4's utility-first approach is a meaningful shift from Bootstrap's component model. The payoff (full control over visual language) is worth it for a custom-designed brand like vasuqi, but the learning curve is real.
- **Spec drift — Story 1-2** — colour token values in `epics.md` were stale vs. the Coolors export. Review caught it, but it caused a re-check pass. The Figma file and `design-tokens.css` are canonical; written planning docs are secondary.
- **Review density on Story 1-3** — 17 review findings on the first real UI story (nav/footer), including 2 WCAG contrast failures. These should have been caught at design time, not code review. Pattern: visual complexity → higher review load.

---

## Key Insights

1. **Epic 1 = vocabulary, Epic 2 = first sentence.** The design system now exists in code. Epic 2 is where it becomes visible.
2. **Figma + `design-tokens.css` are the canonical sources.** When planning docs conflict with visual sources, visual sources win.
3. **Review density tracks visual complexity.** Story 1-3 (first UI) had 3× the findings of 1-1 (pure scaffold). Epic 2 will be denser still.
4. **Accessibility testing belongs in code, not Figma.** Test token pairings as each section is implemented. The palette has been validated — the concern is which tokens go where.
5. **Accessibility fixes should preserve aesthetic intent.** Small overlay opacity nudges are acceptable; replacing palette colours is not. The vasuqi visual identity is non-negotiable.
6. **Animation stories need a different done-definition.** 'Edda reviews in browser' is a required gate, not optional. Motion cannot be spec'd purely in text.
